     Resolution strategy details

     Contact resolution (attendee_email ‚Üí contactId UUID)

     In the extractor, after fetching a batch of checkins:
     SELECT id, email FROM agentcis.clients WHERE email IN (?)
     Build a Map<email, clientId>. The transformer then calls
     idResolver.resolveContactId(clientId) using the existing contacts.json +
     temp_mapped_contacts fallback chain.

     Unmatched-email handling: skip the checkin row (return null from
     transformImpl). Rationale: email matching is inherently lossy ‚ÄĒ walk-in
     attendees may not be clients, emails may be misspelled, etc. The
     orchestrator's Promise.allSettled path already treats null returns as
     skipped (per the prior exploration of migration.orchestrator.ts:270). Log
     the skipped uuid + email so the data team can audit.

     User resolution (host_email ‚Üí assigneeId UUID)

     Same pattern: extractor batch-queries agentcis.users WHERE email IN (?),
     transformer calls resolveUserId. Unmatched-email handling: set assigneeId =
      null (the field is already nullable in ApplyIMSOfficeVisit). This is more
     lenient than contact ‚ÄĒ host being unresolvable just means we lose the
     assignee, but we don't lose the whole visit.

     Branch resolution (office_name ‚Üí branchId UUID)

     Regex /-(\d+)$/ against office_name, then
     resolveBranchId(parseInt(match[1])). Unmatched/missing: throw ‚ÄĒ office_name
      is structured data and a failure here is a real bug, not a data quality
     issue.

     createdBy

     Read directly from config: config.migrationAdminUserId. Validate it's a
     UUID at startup. No per-row lookup.

     - Cursor type: epoch milliseconds (number), stored in the existing
     migration_checkpoints.lastProcessedId (assumed bigint ‚ÄĒ verify in entity).
     - Query:
     SELECT ... FROM checkins
     WHERE old_id IS NULL

     - getTotalCount: SELECT COUNT(*) FROM checkins WHERE old_id IS NULL.

‚óŹ Walking through the SQL step by step.

  What it does: Deduplicates the checkins table by uuid, keeping only the "best"
   row per uuid.

  Step by step:

  1. Innermost SELECT: Pulls every row from agentcis.checkins WHERE old_id IS 
  NULL (same filter we already use ‚ÄĒ only checkins-native data, not back-filled
  from office_visits).
  2. ROW_NUMBER() OVER (PARTITION BY uuid ORDER BY ...): For each group of rows
  that share the same uuid, assigns a rank rn = 1, 2, 3.... The ordering inside
  each partition uses a priority ladder, where each CASE WHEN X IS NOT NULL THEN
   0 ELSE 1 makes non-null values sort before null:

  | Priority | Field                      | Why it matters for the migration
       |
  |----------|----------------------------|-------------------------------------
  -----|
  | 1st      | host_email IS NOT NULL     | Gives us assigneeId
       |
  | 2nd      | completed_time IS NOT NULL | Gives us Completed status +
  sessionEnd   |
  | 3rd      | attended_time IS NOT NULL  | Gives us Attending status +
  sessionStart |
  | 4th      | check_in_time DESC         | Tie-breaker ‚ÄĒ pick the most recent
       |

  3. WHERE rn = 1: Keeps only the top-ranked row per uuid.
  4. Final SELECT * FROM deduped_checkins: Returns one row per uuid ‚ÄĒ the
  "richest" / "most complete" version.

  The implication: This query exists because there are duplicate uuid values in 
  checkins. Our CheckinExtractor currently does not dedupe ‚ÄĒ it just filters
  old_id IS NULL and paginates. If a uuid appears twice, we'd attempt to
  bulk-create it twice in ApplyIMS. The upsert on temp_mapped_checkins would
  collapse the mapping rows fine, but ApplyIMS could end up with duplicate
  visits depending on whether its API dedups by agentcisId.

  The intent of the SQL maps directly to what the migration cares about: pick
  the row that yields the most useful target record (assignee, full status,
  latest check-in).

‚óŹ Honestly? No, it's a sketchy design choice. I made the call to get the feature
   working but the user is right to push back. Here's why it bothers me:

  Problems with the current approach (walk-in creation inside the extractor):

  1. ETL boundary violation. Every other extractor in this codebase is read-only
   against AgentCIS. This one now does API writes to ApplyIMS during extract.
  Anyone reading checkin.extractor.ts would not expect that ‚ÄĒ the name lies
  about what it does.
  2. Bypasses the standard error pipeline. When a "real" entity hits an API
  error, it flows through BatchProcessor ‚Üí ErrorRecoveryManager ‚Üí
  migration_errors table with category tagging (API_ERROR, NETWORK_ERROR, etc.).
   My in-extractor bulkCreateContacts call has none of that ‚ÄĒ failures throw raw
   out of the batch and aren't recorded in migration_errors.
  3. Bypasses rate limiting & retry. The ApiClient does have rate limiting built
   in, but the orchestrator's batch-pacing assumes one bulk API call per batch
  (the entity's main create). We're now making two: walk-in contacts + office
  visits.
  4. No mapping persistence. Every rerun re-POSTs all walk-ins. ApplyIMS's
  DUPLICATE_EMAIL path recovers the UUIDs, so it's not broken ‚ÄĒ just wasteful.
  The existing temp_mapped_* pattern would have saved this work.
  5. Optional-deps smell. Constructor takes three ? params. When the test route
  omits them, walk-in creation silently skips and rows turn into nulls. Same
  code, different behavior depending on construction site.
  6. Bypasses the orchestrator's Promise.allSettled boundary. A walk-in API
  failure now kills the whole extractBatch call, which kills the orchestrator's
  batch. With the standard pipeline, one bad record skips and the rest continue.

  Cleaner alternative: Add a pre-flight step in the orchestrator ‚ÄĒ same pattern
  as populateDealStaging (which runs after CONTACTS migration). It would:
  - Run before CHECKINS migration begins
  - Scan checkins once for emails missing from clients
  - Bulk-create those contacts through the existing BatchProcessor (gets error
  logging, retries, rate limiting for free)
  - Persist mappings in a new temp_mapped_walkin_contacts(email, 
  applyims_contact_id) table
  - Then the extractor stays read-only and looks up walk_in_contacts by email as
   a third fallback after clients

  Cost of the cleanup: new entity + new DB migration + new repo method + new
  orchestrator method + new IdResolver lookup or similar. ~5 files. The
  CheckinExtractor sheds its API-client dependency.

     Move walk-in contact creation out of CheckinExtractor

     Context

     The current implementation creates ApplyIMS "walk-in" contacts inline
     inside CheckinExtractor.extractBatch() whenever attendee_email doesn't
     match an agentcis.clients row. This works but violates the codebase's ETL
     boundary in several ways:

     ApplyIMS ‚ÄĒ surprising and unsafe.

     3. Bypasses BatchProcessor's retry + rate-limit handling.

     5. Three optional constructor params mean the test route behaves
     differently from production.
     6. A walk-in API failure kills the whole extractBatch, escaping
     Promise.allSettled at the orchestrator boundary.

      pattern as populateDealStaging, which runs after CONTACTS). After the
     refactor:
     - CheckinExtractor is read-only again.
     - Walk-in contacts are created once, before the CHECKINS entity runs, and
     persisted in a new mapping table.
     - The extractor's email lookup gains a 3rd source: the walk-in mapping

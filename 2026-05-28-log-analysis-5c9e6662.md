Source: `./logs/migration-2026-05-{27,28}.log*` and `./logs/error-2026-05-{27,28}.log*`
Project: agentcis → applyims ETL (Phase 2), staging environment.

## Summary

- Single long-running migration that **started 2026-05-27 12:43:58 UTC, crashed 2026-05-27 22:08:35 UTC** (~9h25m wall time), then was resumed twice on 2026-05-28 (10:56 and 11:08), with the last logged activity at 11:22:23. The active uncompressed log (`migration-2026-05-28.log.17`) cuts off mid-attachments — the run never reached a clean "Migration completed" event for this id.
- **Crash root cause:** during the checkins phase's walk-in contact pre-flight, the orchestrator inserted a row into `temp_mapped_walkin_contacts` with a NULL `applyims_contact_id`, violating the NOT NULL constraint. The pre-flight is in compiled `dist/orchestrators/migration.orchestrator.js:494-510` but **not present in `src/`** — the source has drifted from the running build.
- **~1.46M total error records** for this migration id (1,431,063 transformation + 32,619 API + 6 validation). Most are downstream cascades from a small number of upstream gaps (applications + notes + contacts).
- A **1h41m log gap** between 18:14:59 and 19:56:01 hides the end-of-applications, the entire `notes` entity stage, and the start of `contact-activities`. Notes never appears in the migration lifecycle stream — only the error log betrays that it ran.
- A previously-undocumented entity type, **`notes`** (between applications and contact-activities), produced 131,460 `Cannot resolve applicationId for agentcis application <id>` errors — the largest single error class after contact-activities.

## Per-entity batch timeline highlights (deep dive)

**contacts** (392,663 records, 786 batches): 739 batches contained at least one failure; mostly `Invalid email: null` (3,739) + duplicate-key on `IDX_LOWERCASE_EMAIL` (29,484) — these are split between transform-time drops and API-time rejections, with the duplicate-key wave indicating the target DB already held those emails (idempotency gap on re-runs).

**applications** (511 batches, ~38% success): the first batch landed 255/500 successes at 14:48:51. By batch 6 (15:07:51) the success count hit zero and stayed mostly at zero for the rest of the run. Behavior alternated between two regimes:
- "0 successful / ~280 failed" → dealId/contactId resolution miss (transformation cascade).
- "0 successful / 0 failed" → batch silently produced no output (transformer returned null) — likely the createdAt null edge case.

281 of 511 batches recorded `successful=0`. There are brief recoveries (e.g. batches 51-100 ran at 470+ successes per 500) when the dealId map was warm.

**contact-activities** (4,022 batches, started in log gap): on entry to the captured window (19:56:01, 13.38% done), batches were producing only **45-318 successes per 500** (9-64% — most batches sub-30%). Failure count is always 0 here because the transformer `throw`s before reaching the API loader, and the BatchProcessor records those as drops in the error log but not as batch-level "failed".

**office-visits** (1,570 batches): healthy, ~95-100% success per batch throughout. 2,468 API_ERROR rows attributed to FK violation on `contact_visits.FK_…fae` — same underlying issue (missing contact mapping).

**checkins** (216 batches on resume 2, 97.68% processed): only 12 batch-level failures across the whole stage. 1,004 transform drops with `attendee email is not a registered client` — these are walk-ins for which the pre-flight failed in the 5-27 run AND were not retried in the 5-28 resume because the temp table no longer has the mapping.

**attachments** (195 batches before log cutoff): 90,349 successes, 0 batch-level failed, 13.16% processed. 3,968 transform drops on `Cannot resolve client ID for application <id>` (depends on applications mapping) and 3,058 + 125 + 89 + 2 drops on unsupported `attachmentableType` values (email/partner/task/product) — the latter are deliberate skips per the transformer, not bugs.

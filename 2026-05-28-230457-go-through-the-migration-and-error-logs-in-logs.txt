  The contact transformer treats user fields as nullable:

  - assigneeId, archivedBy ‚ÄĒ left null if the user ID can't be resolved

  4. API-side duplicate key on email (29,484 records)
  5. API-side duplicate key on contact_follower_uniq_idx (103 records)

  independent of the missing-92-users problem.

  The cost is silent data quality degradation in the contacts that did migrate:
  NULL createdBy/assigneeId/archivedBy and shorter followers lists ‚ÄĒ not lost
  records.

  Two fixes

  Fix 1 (narrow, surgical) ‚ÄĒ add a JSON-aware exclusion to populateDealStaging's
   SQL. Load the applications.json agentcis_id set on orchestrator startup; pass
   to the query as AND app.id NOT IN (:priorMigratedAppIds). Buckets become
  smaller (only contain apps that will actually be POSTed this run); fewer
  redundant dealIds get generated; no orphan deals.

  This is correct for "JSON apps already belong to a prior deal we don't touch,"
   but it does not re-associate prior apps with the new deal even if your
  business intent is "all apps in this 6-month window should share one deal."
  That requires Fix 2.

  Fix 2 (broader) ‚ÄĒ re-associate JSON apps with the new deal. During
  populateDealStaging, for buckets that include JSON apps, instead of generating
   a fresh dealId, look up the prior deal via the JSON app's existing dealId
  (which would require applications.json to include deal_id, or a separate
  deals.json). Then either reuse that dealId for the bucket, or PATCH the new
  apps to point to the prior deal.

  This is a much bigger change ‚ÄĒ the JSON mapper format doesn't currently carry
  deal_id, the ApplyIMS API may not expose deal-update endpoints, and the
  bucketing logic may not align with the prior migration's deal grouping.
  Probably out of scope for a hotfix.

  Recommendation for this codebase: Fix 1 is the right immediate change. Combine
   it with adding a per-batch skipped field to the progress event (it already
  exists in the final summary at orchestrator.ts:196, just needs to be added to
  the periodic progress emit) so operators can distinguish "0/500 success
  because all skipped" from "0/500 success because all failed."

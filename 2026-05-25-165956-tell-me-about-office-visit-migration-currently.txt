  How the Transformer Works

  - Resolves contactId, assigneeId (nullable), branchId (from officeId),
  createdBy (from userId) via IdResolver
  - Maps visitPurpose ‚Üí sessionNotes
  - Gets sessionStart/sessionEnd from officeVisitAssignees via LEFT JOIN in
  extractor
  - Computes status: sessionStart && sessionEnd ? 'Completed' : 'Unattended'
  - Pulls visitPurposeId from global config

  Known Limitations

  1. Status logic is simplified ‚ÄĒ Only produces 'Completed' or 'Unattended'.
  ApplyIMS supports 5 statuses (Pending, Waiting, Attending, Completed,
  Unattended), but intermediate states are not mapped.
  2. visitPurposeId is hardcoded from config ‚ÄĒ no validation that it actually
  exists in ApplyIMS.
  3. Null assignee is allowed but not explicitly validated downstream.

  Overall Assessment

  The implementation is structurally complete and integrated into the migration
  flow. It will run as part of runMigration() in the correct dependency order.
  The main simplification is the status mapping ‚ÄĒ only two of five possible
  statuses are produced, which may be acceptable depending on how AgentCIS
  stores visit state.
